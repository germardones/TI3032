4. Formatos: Localidad de Datos, JSON y BSON

En el almacenamiento Relacional estricto (SQL), la regla inquebrantable de la arquitectura obligaba a "Normalizar". Cada elemento repetido o foráneo debía abstraerse y dividirse en tablas secundarias.

En el universo de Bases de Datos NoSQL y Documentales, las reglas cambian hacia la asimetría para priorizar inmediatez: "Los datos que se acceden juntos en la aplicación, deben guardarse juntos estáticamente". A esto en Big Data se lo categoriza como Localidad de Datos.

El Subdocumento Anidado (Agregado)

{
  "usuario": "aperez",
  "rol_metadata": {
    "permiso": "sudo_admin",
    "frecuencia_login": "daily"
  },
  "dispositivos_registrados": [
    { "os": "Android", "activo": true },
    { "os": "Win11", "activo": false }
  ]
}

Una sola llamada a disco obteniendo este documento trae todos sus roles en el Subdocumento y su Array de Dispositivos simultáneamente. Cero JOINs y se elimina la sobrecarga computacional de fusionar 3 tablas para cargar un dashboard web.

Bajo el Capó: El paso de JSON al formato Binario (BSON)

Es un error común pensar que un motor como MongoDB almacena e indexa simplemente objetos JSON en disco duro.

El formato JSON (JavaScript Object Notation) domina la web por ser ilegible universal, pero es limitado: carece de noción pura de longitudes y de tipos de datos maduros (no permite definir binarios de archivos crudos dentro del código ni tiene control hiper-preciso sobre fechas en lugar de strings opacas).

Por eso el motor toma y codifica permanentemente el texto dictado en JSON convirtiéndolo a BSON (Binary JSON). BSON inyecta nativamente los tipos robustos faltantes (como su famoso e insustituible autogenerador ObjectId) y, de forma crítica, introduce Prefijos de Longitud Dinámicos y Empaquetado Binario.

  • Eficiencia en Memoria C: Pointer Jumping (Saltos por Punteros)
    Si el indexador interno necesita revisar la llave "estado" al final del documento, el formato BSON le permite consultar el empaquetado inicial que contiene el tamaño exacto del documento saltando los bytes iniciales innecesarios, decodificando exclusivamente lo necesario sin tener que inflar RAM (ni convertir a string) toda la estructura anterior del documento.

Tipos de Datos en BSON

Gracias a BSON, MongoDB puede diferenciar y operar nativamente con una rica gama de tipos, extendiendo los de JSON básico. A continuación se presentan todos los tipos disponibles con ejemplos de uso:

String

Cadena de Texto

Almacena texto en codificación UTF-8. Es el tipo más común: nombres, descripciones, correos, estados.

"nombre": "Ana Rodríguez",
"email":  "ana@inacap.cl",
"estado": "activo"

En $jsonSchema usa bsonType: "string"

Int32

Entero de 32 bits

Número entero entre −2 147 483 648 y 2 147 483 647. Ideal para contadores, edades y cantidades moderadas.

"edad":     28,
"intentos": 3,
"stock":    150

En $jsonSchema usa bsonType: "int"

Int64 / Long

Entero de 64 bits

Para valores enteros muy grandes (hasta ±9.2 × 10¹⁸). Usado en IDs numéricos externos, timestamps Unix o contadores de alta escala.

"visitas_totales": NumberLong("5200000000"),
"id_externo":     NumberLong("9007199254740993")

En $jsonSchema usa bsonType: "long"

Double

Número de Punto Flotante

Número real de 64 bits (IEEE 754). Útil para porcentajes, coordenadas GPS y métricas. Puede acumular error de redondeo en cálculos financieros.

"latitud":   -33.4489,
"longitud":  -70.6693,
"promedio":  6.75

En $jsonSchema usa bsonType: "double"

Decimal128

Decimal de Alta Precisión

128 bits de precisión decimal. Obligatorio en aplicaciones financieras, contables o científicas donde el error de redondeo de Double es inaceptable.

"precio":   NumberDecimal("19990.50"),
"descuento": NumberDecimal("0.125")

En $jsonSchema usa bsonType: "decimal"

Boolean

Valor Lógico

Solo admite los literales true o false. Se usa para flags, activaciones y estados binarios. No usar strings "true"/"false".

"activo":        true,
"email_verificado": false,
"es_admin":       true

En $jsonSchema usa bsonType: "bool"

Date

Fecha y Hora (ISODate)

Almacena instantes de tiempo como un entero de 64 bits firmado con los milisegundos transcurridos desde el Epoch Unix (1 enero 1970 UTC). No depende de formatos string, por lo que las consultas de rango, ordenamiento y aritmética de fechas son nativas y eficientes.

INSERTAR fechas

// Shell de MongoDB
"creado_en": new Date(),
"nacimiento": ISODate("2000-06-15"),
"vence_en": ISODate("2026-12-31T23:59:59Z")

CONSULTAR con rangos

// Documentos del último mes
db.logs.find({
  "creado_en": {
    $gte: ISODate("2026-03-01"),
    $lt:  ISODate("2026-04-01")
  }
})

EXTRAER partes de fecha (aggregation)

{ $year:  "$creado_en" }  → 2026
{ $month: "$creado_en" }  → 4
{ $dayOfWeek: "$creado_en" } → 4
{ $hour: "$creado_en" }   → 14

ERROR COMÚN a evitar

// MAL: string opaco, no ordenable
"fecha": "2026-04-23"

// BIEN: tipo Date nativo
"fecha": ISODate("2026-04-23")

En $jsonSchema usa bsonType: "date"  |  El motor guarda ISODate("2026-04-23") como el entero 1745366400000 ms internamente.

ObjectId

Identificador Único

12 bytes: 4 bytes de timestamp + 5 de aleatoriedad del proceso + 3 de contador incremental. Auto-asignado al campo _id. Lleva la fecha de creación embebida.

"_id": ObjectId("6625f3a2c4e1d80012ab34cd")

// Extraer timestamp de creación
ObjectId("6625f3a2...").getTimestamp()

En $jsonSchema usa bsonType: "objectId"

Array

Arreglo Ordenado

Lista de valores del mismo u otro tipo. Permite almacenar múltiples valores en un campo y consultarlos con operadores específicos ($in, $all, $elemMatch).

"tags": ["mongodb", "bson", "nosql"],
"notas": [6.5, 5.8, 6.9],
"historial": [
  { "accion": "login", "ok": true }
]

En $jsonSchema usa bsonType: "array"

Object

Subdocumento Embebido

Un documento anidado con sus propios campos. Se accede mediante dot notation. Implementa el patrón de Localidad de Datos sin JOINs.

"direccion": {
  "calle":   "Av. Irarrázaval 1571",
  "ciudad":  "Santiago",
  "comuna":  "Ñuñoa"
}
// Consulta: "direccion.ciudad": "Santiago"

En $jsonSchema usa bsonType: "object"

Null

Ausencia de Valor

Representa explícitamente la ausencia o indeterminación de un valor. Diferente a omitir el campo: permite consultar documentos donde el campo existe pero no tiene dato.

"segundo_apellido": null,
"foto_perfil":      null

// Buscar campos nulos
{ "foto_perfil": null }

En $jsonSchema usa bsonType: "null"

BinData

Datos Binarios Crudos

Almacena bytes arbitrarios: imágenes, archivos, hashes criptográficos o UUIDs. Evita el overhead de codificación Base64 que impondría JSON puro.

"avatar": BinData(0, "iVBORw0KGgoAAAA..."),
"uuid":   BinData(4, "ESIziETGeUCblA...")

// Subtipos comunes: 0=genérico, 4=UUID

En $jsonSchema usa bsonType: "binData"

Timestamp

Marca de Tiempo Interna

Tipo especial de 64 bits reservado para uso interno del motor de replicación de MongoDB (oplog). No debe usarse para fechas de negocio — para eso existe Date.

// Aparece en el oplog de réplicas
"ts": Timestamp(1745366400, 1)
//            ^epoch-seg  ^ordinal

En $jsonSchema usa bsonType: "timestamp"

Regex

Expresión Regular

Almacena un patrón de expresión regular PCRE directamente como tipo nativo. Permite búsquedas de texto flexible con $regex, aunque en colecciones grandes se prefiere un índice de texto completo.

// Buscar emails que terminen en .cl
db.usuarios.find({
  "email": { $regex: /\.cl$/i }
})

En $jsonSchema usa bsonType: "regex"

MinKey MaxKey

Claves de Límite

Valores especiales que se comparan como menor que todo (MinKey) o mayor que todo (MaxKey) en el orden de comparación BSON. Usados en fragmentación (sharding) para definir rangos de partición.

// Shard chunk boundaries
{ "min": MinKey(), "max": "M" }
{ "min": "M",       "max": MaxKey() }

Internamente tipo 0xFF (MinKey) y 0x7F (MaxKey)
