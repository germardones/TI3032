1. El Cambio de Paradigma: SQL vs NoSQL

Durante décadas, las bases de datos relacionales (SQL) fueron el estándar inamovible de la industria para persistir información. Su fortaleza radicaba en la normalización: dividir la información en múltiples tablas conectadas mediante llaves foráneas para evitar redundancia y garantizar integridad (ACID).

Sin embargo, en el tránsito hacia la web moderna (Web 2.0 y 3.0), el surgimiento del Big Data, Redes Sociales y el Internet de las Cosas (IoT) provocó tres problemas arquitectónicos masivos:

  • Volumen: Cantidades de datos en orden de Petabytes, imposibles de guardar en un solo disco físico.
  • Velocidad: Millones de operaciones de lectura/escritura por segundo (ej. un sensor midiendo telemetría industrial cada milisegundo).
  • Variedad: Datos que no tienen una estructura rígida prefijada (imágenes cruzadas con diccionarios de metadatos variables).

Para resolver esto emergió la arquitectura NoSQL (Not Only SQL). No reemplaza a SQL, pero brinda respuestas distribuidas y dinámicas cuando las tablas rígidas se transforman en un cuello de botella infranqueable.

Modelo Tradicional Estructurado (SQL)

  • Mecanismo de Guardado: Tablas Rígidas.
  • Esquema: Schema-On-Write (Se debe crear la tabla y definir tipos de datos rígidos [INT, VARCHAR] antes de insertar).
  • Consultas: Fuertemente dependiente de sentencias JOIN en memoria para unir datos. Lento para cargas exponenciales.
  • Ejemplos Clásicos: PostgreSQL, MySQL, Oracle.

Modelo Moderno No Estructurado (NoSQL)

  • Mecanismo de Guardado: Documentos JSON, Clave-Valor, Grafos.
  • Esquema: Schema-On-Read (Esquema Dinámico, el documento guarda lo que le mandes sin validación rígida previa de columnas).
  • Consultas: Favorece guardar la información junta en vez de desintegrarla, invalidando la necesidad de los lentos JOINs computacionales.
  • Ejemplos Clásicos: MongoDB, Redis, Cassandra.
