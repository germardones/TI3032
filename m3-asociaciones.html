3. Asociaciones Lógicas: Subdocumentos y Arreglos

En SQL las relaciones entre entidades se implementan mediante claves foráneas (FOREIGN KEY) y JOINs en tiempo de consulta. MongoDB adopta una filosofía distinta: permite incrustar documentos relacionados directamente dentro de un documento padre. A esta técnica se le llama embedding o subdocumentos. Comprender cuándo aplicar cada estrategia es uno de los skills de diseño más críticos en NoSQL.

Las Dos Estrategias Fundamentales

Estrategia 1: Embebido (Embedding)

  • El documento relacionado vive dentro del padre como objeto anidado o elemento de un arreglo.
  • Una sola operación find() trae todo el árbol de datos: no hay JOIN.
  • Óptimo cuando los datos relacionados se leen juntos casi siempre.
  • El documento hijo no tiene identidad propia en el sistema.

Estrategia 2: Referenciado (Referencing)

  • El hijo vive en su propia colección. El padre guarda solo el _id del hijo como referencia manual.
  • Se requieren múltiples queries para unir los datos.
  • Óptimo para entidades independientes, alta cardinalidad o datos actualizados frecuentemente de forma autónoma.
  • El documento hijo tiene identidad y ciclo de vida propios.

Patrón 1:1 — Subdocumento Embebido

Cuando una entidad tiene una relación de uno a uno con otra (por ejemplo, un usuario y su dirección) y siempre se acceden juntos, el embedding es la elección correcta. El resultado es una sola lectura de disco para obtener todos los datos.

Colección: usuarios — Dirección embebida (1:1)

db.usuarios.insertOne({
  _id:    ObjectId("507f1f77bcf86cd799439011"),
  nombre: "Valentina Ríos",
  email:  "vrios@example.com",

  // Subdocumento embebido — no es una colección separada
  direccion: {
    calle:         "Av. Providencia 1234",
    ciudad:        "Santiago",
    region:        "Metropolitana",
    codigo_postal: "7500000"
  }
})

La direccion no es una referencia: es un objeto JSON anidado incrustado directamente. Cuando MongoDB lee este documento con un solo find(), la dirección viene incluida sin costo adicional de red ni de CPU para ejecutar un JOIN.

Patrón 1:N — Arreglo de Subdocumentos Embebidos

Cuando una entidad padre tiene múltiples entidades hijas (un pedido con varios ítems) y esos ítems son parte inherente del pedido, se pueden incrustar como un arreglo de subdocumentos.

Colección: pedidos — Items embebidos como arreglo (1:N)

db.pedidos.insertOne({
  _id:        ObjectId("507f191e810c19729de860ea"),
  cliente_id: ObjectId("507f1f77bcf86cd799439011"),
  fecha:      "2026-04-14",
  estado:     "procesando",
  total:      125900,

  // Arreglo de subdocumentos — cada ítem es un objeto anidado
  items: [
    {
      producto_nombre: "Teclado Mecánico",
      cantidad:        1,
      precio_unitario: 85000
    },
    {
      producto_nombre: "Mouse Inalámbrico",
      cantidad:        2,
      precio_unitario: 20450
    }
  ]
})

*En SQL esto requeriría dos tablas (pedidos y pedido_items) con FOREIGN KEY y un JOIN para recuperar los datos completos. En MongoDB todo reside en un solo documento, en un solo bloque de disco, leído en una sola operación de I/O.*

Patrón 1:N — Referencia por ObjectId

Cuando el documento hijo tiene vida propia (se consulta de forma autónoma, se actualiza frecuentemente o pertenece a múltiples padres), se guarda en su propia colección y el padre almacena solo su _id. Este patrón equivale a la clave foránea de SQL.

Colección: autores

db.autores.insertOne({
  _id:    ObjectId("a1b2c3"),
  nombre: "Gabriel Morales",
  pais:   "Chile",
  bio:    "Escritor y..."
})

Colección: libros (con referencia)

db.libros.insertOne({
  _id:    ObjectId("d4e5f6"),
  titulo: "El Código del Algoritmo",
  year:   2025,
  // Referencia manual al ObjectId del autor
  autor_id: ObjectId("a1b2c3")
})

Nota importante: MongoDB no impone integridad referencial automática. Si eliminas el documento de un autor, los libros que lo referencian no se eliminan en cascada — quedarán con un autor_id huérfano. La integridad referencial debe implementarse a nivel de aplicación.

Patrón N:N — Arreglo de Referencias

Las relaciones muchos-a-muchos se modelan guardando un arreglo de ObjectIds en uno o ambos lados de la relación.

# Estudiante con arreglo de IDs de cursos inscritos
db.estudiantes.insertOne({
  _id:    ObjectId("est001"),
  nombre: "Carlos Fuentes",
  // Arreglo de referencias a la colección "cursos"
  cursos_inscritos: [
    ObjectId("curso_bd_nosql"),
    ObjectId("curso_python"),
    ObjectId("curso_docker")
  ]
})

Marco de Decisión: ¿Embeber o Referenciar?

Criterio Embeber Referenciar
Patrón de acceso Siempre se leen juntos Se consultan por separado frecuentemente
Cardinalidad Pocos hijos (1:1, 1:few) Muchos hijos (1:N, N:N)
Ciclo de vida El hijo depende del padre El hijo existe de forma autónoma
Frecuencia de actualización El hijo rara vez cambia El hijo se actualiza de forma independiente
Tamaño del documento No supera el límite de 16 MB Riesgo de superar 16 MB sin referencias

Regla de oro: Diseña tu esquema en función de cómo tu aplicación lee los datos, no de cómo se relacionan conceptualmente. Si el 90% de las queries necesitan un usuario con su dirección juntos, embeberla es correcto. Si frecuentemente consultas direcciones sin usuarios, la dirección merece su propia colección.
