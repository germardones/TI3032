2. Creación y Eliminación de Bases de Datos y Colecciones

Una de las diferencias más radicales de MongoDB respecto a los RDBMS es su modelo de creación implícita diferida: ni las bases de datos ni las colecciones se crean con sentencias DDL explícitas como CREATE DATABASE o CREATE TABLE. MongoDB las instancia en memoria y las persiste en disco en el momento en que recibe el primer documento. Este enfoque prioriza la agilidad en el desarrollo sobre la rigidez estructural.

El Comando use — Selección y Creación Implícita

El comando use <nombre> apunta el contexto del shell hacia una base de datos específica. Si la base de datos no existe todavía, MongoDB la reserva en memoria pero no la escribe a disco hasta que se inserte el primer documento en alguna de sus colecciones.

# Seleccionar (o crear implícitamente) una base de datos
use tienda_online
// → switched to db tienda_online

# Verificar en qué base de datos estás actualmente
db
// → tienda_online

# Listar las bases de datos persistidas en disco
show dbs
// ⚠ tienda_online NO aparece aquí todavía — está vacía.
// admin    40.00 KiB
// config  108.00 KiB
// local    72.00 KiB

Comportamiento clave: show dbs solo lista bases de datos que contienen al menos una colección con al menos un documento. Una DB vacía existe únicamente en el contexto del shell. Este es un mecanismo de lazy evaluation a nivel de sistema de archivos.

Creación de Colecciones — Implícita y Explícita

Las colecciones también se crean automáticamente al insertar el primer documento. Sin embargo, el comando db.createCollection() permite crearlas con opciones avanzadas de configuración antes de insertar datos.

Creación Implícita (Más Común)

# La colección "productos" se crea
# automáticamente en este insert
db.productos.insertOne({
  nombre: "Teclado Mecánico",
  precio: 85000
})

# Verificar que fue creada
show collections
// → productos

Creación Explícita con Opciones

# Crear con validación de esquema JSON
db.createCollection("usuarios", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["email", "nombre"],
      properties: {
        email: { bsonType: "string" },
        edad:  { bsonType: "int",
                 minimum: 18 }
      }
    }
  }
})

Colecciones Limitadas (Capped Collections)

Un tipo especial de colección con tamaño fijo en bytes y orden de inserción garantizado. Cuando alcanza el límite, sobreescribe automáticamente los documentos más antiguos (como un buffer circular). Son ideales para logs, colas de eventos y streams de datos en tiempo real.

# Crear colección capped de máximo 5 MB y 1000 documentos
db.createCollection("logs_sistema", {
  capped: true,
  size:   5242880,   // 5 MB en bytes (obligatorio)
  max:    1000       // Límite adicional de documentos (opcional)
})

# Verificar que fue creada como capped
db.logs_sistema.isCapped()   // → true

*En colecciones capped no se pueden eliminar documentos individuales ni actualizar documentos de forma que aumenten su tamaño. Estas restricciones garantizan el orden FIFO del buffer circular.*

Inspeccionando la Estructura Actual

Estos comandos son fundamentales durante el desarrollo y la administración para auditar qué estructuras existen en el servidor:

# Listar todas las bases de datos del servidor
show dbs

# Listar todas las colecciones de la DB activa
show collections

# Ver estadísticas detalladas de una colección
db.productos.stats()
// → ns, count, size, avgObjSize, storageSize, totalIndexSize ...

# Contar documentos en una colección
db.productos.countDocuments()

Eliminación de Colecciones — drop()

El método drop() elimina permanentemente una colección: todos sus documentos, índices y configuración de validación. La operación es atómica e irreversible sin un backup previo.

Operaciones de Eliminación — Sin Reversa

# Eliminar UNA colección específica
db.productos.drop()       // → true (si existía) | false (si no)

# Verificar que ya no existe
show collections          // "productos" ya no aparece

# Eliminar TODA la base de datos activa (¡Operación nuclear!)
db.dropDatabase()         // → { ok: 1, dropped: "tienda_online" }

La llamada a db.dropDatabase() destruye permanentemente todas las colecciones y documentos de la base de datos activa. En producción esta operación requiere permisos de rol dbAdmin o dbOwner y jamás debe ejecutarse sin una copia de seguridad verificada.

Renombrar una Colección

MongoDB permite renombrar colecciones sin mover físicamente los datos. La operación solo actualiza los metadatos del catálogo interno; todos los documentos e índices se conservan intactos.

db.productos.renameCollection("catalogo_productos")
// La colección ahora se llama "catalogo_productos"
// Todos los documentos e índices se conservan intactos
